# AGENTS.md — dagger-ci

See [flo-docs/AGENTS.md](https://github.com/Flomenco-Inc/flo-docs/blob/main/AGENTS.md)
for shared conventions. This file is the repo-local supplement.

## Module overview

Thin Dagger wrapper around the three CI jobs every flo Terraform module runs
(see README.md for user-facing docs). All logic lives in `main.go`; there is
no package layering. Keep it that way until this file grows past ~500 lines —
then split by function family.

## File organization

- `main.go` — module entry point + all exported functions + internal helpers.
- `dagger.json` — Dagger module manifest (name, engine version, SDK).
- `go.mod` — declares Go version + `dagger.io/dagger` at the matching engine
  version. Regenerated by `dagger develop`; commit only the minimal form
  (module name + go version + dagger dep).
- `internal/dagger/` — **generated, not committed.** Produced by
  `dagger develop`. Contains SDK bindings scoped to this module's dagger.json.
  Listed in `.gitignore`.

## Tool-version pins

The constant block at the top of `main.go` is the single source of truth for
Terraform / TFLint / terraform-docs / Checkov / Python versions. DO NOT hard-
code versions inside helper functions. DO NOT add a `versions.yaml` or
`renovate-baserules.json`. The `const ( ... )` block is intentionally the
simplest possible mechanism and should stay that way.

## What NOT to do

- **Do not add AWS credential handling here.** This module is CI-only
  (fmt, validate, lint, scan). Anything that calls AWS APIs belongs in
  `dagger-terragrunt`. If you find yourself needing `*dagger.Secret` for AWS
  creds in this repo, stop — you are in the wrong module.
- **Do not add test runners for application code.** No Go test, no pnpm test,
  no pytest. Those are Turborepo's concern in the `flo` monorepo.
- **Do not couple this module to flo-specific paths.** Functions take a
  `*Directory` parameter for a reason — the module must work on any Terraform
  source, not just flo's. If you find yourself writing `/src/terraform-aws-*`
  or similar, rewrite the function to take the narrowing path as an argument.
- **Do not commit `internal/dagger/` or `dagger.gen.go`.** They are
  engine-version-specific generated output. `.gitignore` excludes them; keep
  it that way.
- **Do not add a rolling `@v0` tag.** Dagger modules resolve Git refs and a
  floating tag would propagate upstream breakage to every consumer
  simultaneously. Consumers pin exact `@vX.Y.Z` versions, and Renovate bumps
  them.
- **Do not publish new functions that silently access the host network
  without being declared in the README.** The whole point of Dagger is
  hermetic execution. A function that `curl`s an arbitrary URL at runtime
  (beyond tool downloads during container build) breaks reproducibility.

## Local testing

Requires Dagger CLI + Docker daemon. If neither is set up locally, CI runs
the full module smoke test on every push — you can iterate via CI if needed.

```bash
dagger develop                # regenerate internal/dagger/
dagger functions              # lists all exposed functions
dagger call terraform-verify --src=../terraform-aws-network
dagger call checkov --src=../terraform-aws-network export --path=./out.sarif
```

If `dagger develop` errors with "cannot find dagger.io/dagger", verify that
`go.mod` has `require dagger.io/dagger vX.Y.Z` matching `dagger.json`'s
`engineVersion`.

## Versioning

Semver-ish:

- Patch bump: tool version change in the `const (...)` block.
- Minor bump: new function or new arg with `+default` preserving call-site
  compatibility.
- Major bump: breaking arg change or function removal. Announce in
  `flo-docs/LEARNINGS.md` with a migration note.

Tag releases with `git tag v0.1.0 && git push --tags`. CI is expected to gate
tag creation on a green build eventually — for now it's manual.
